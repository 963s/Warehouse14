# Warehouse14 mobile — platform architecture

**One Expo app, both platforms, one shared brain.** iOS and Android are NOT
forked into two codebases — that would lose the 80% shared logic (the
api-client, the surfaces, the theme, the fiscal spine). Instead the shared
core lives once under `src/`, and platform-specific behaviour is isolated to
`.ios.tsx` / `.android.tsx` / `.native.tsx` files ONLY where the two platforms
genuinely diverge (camera, scanner, haptics). Everything else is identical and
connects to the same `api-cloud` server + the same fiscal/TSE backend as the
desktop cashier.

```
apps/mobile/
├─ src/                        # SHARED core — 100% of the business logic
│  ├─ app/                     # expo-router routes (every screen, both platforms)
│  ├─ warehouse14/             # the domain/UI/logic layer (theme, api, surfaces, …)
│  │  ├─ theme.ts              #   design tokens (the official store system)
│  │  ├─ api.ts                #   the authenticated api-cloud client
│  │  ├─ sell/                 #   the cashier spine (cart, finalize, receipt)
│  │  └─ ui/                   #   shared primitives (Card, SectionCard, motion)
│  └─ components/ui/           # shadcn/RNR primitives (Card, Button, Input, …)
├─ ios/                        # native iOS project (generated by expo prebuild)
├─ android/                    # native Android project (generated by expo prebuild)
├─ app.json                    # Expo config (bundle id de.warehouse14.mobile, both)
├─ global.css                  # NativeWind v4 tokens (the official store system)
└─ eas.json                    # EAS build profiles (sim / device / preview / prod)
```

## Platform-specific files (the ONLY places iOS and Android differ)

React Native + expo-router resolves a `.ios.tsx` / `.android.tsx` / `.native.tsx`
variant automatically at import time. These exist solely where a capability
truly differs between the platforms:

| File | Why it diverges |
|---|---|
| `src/warehouse14/CapturePhotoScreen.native.tsx` | Native camera capture (the web export uses a file input). |
| `src/warehouse14/ScanScreen.native.tsx` | Native barcode/QR camera scanner (the web export uses an input). |

Everything else — auth/PIN, the cashier (Verkauf/Ankauf/Kasse), inventory
(Lager), the theme, the API client, the fiscal finalize, the notifications,
the dashboard — is one shared implementation for both platforms.

## How to develop each platform

```bash
cd apps/mobile

# Start Metro (the bundler) — serves BOTH platforms from the one bundle.
pnpm dev                   # = expo start

# Run on a device/simulator from the one source tree:
pnpm ios                   # = expo run:ios     (needs Xcode + a booted simulator)
pnpm android               # = expo run:android (needs Android SDK + an emulator)

# If the native projects drifted (a new native dependency), regenerate them:
pnpm prebuild:clean        # = npx expo prebuild --clean  (regenerates ios/ + android/)
```

### Dev device → local backend

The dev device needs to reach the local api-cloud (`http://localhost:3001` or
the dev Mac's LAN IP). For an Android emulator / USB device use `adb reverse`
so `localhost` on the device maps to the host:

```bash
pnpm adb                   # maps ports 9090 / 3000 / 9001 / 8081 device→host
```

iOS Simulator reaches `localhost` directly. For a physical iOS device, set
`EXPO_PUBLIC_API_BASE_URL` to the dev Mac's LAN IP (see `.env`).

## How to build each platform

Both platforms build from the SAME source via EAS (`--local` = no cloud account
needed; uses your local Xcode / Android SDK):

```bash
# iOS — simulator, device, preview, production profiles (eas.json)
pnpm build:ios:sim
pnpm build:ios:device
pnpm build:ios:preview
pnpm build:ios:prod

# Android — same four profiles
pnpm build:android:sim
pnpm build:android:device
pnpm build:android:preview
pnpm build:android:prod
```

Bundle id (both platforms): `de.warehouse14.mobile`. See `BUILD.md` for signing
(certificates, provisioning profiles, the EAS cloud path).

## Connection to the rest of the system

- **api-cloud** (`https://api.warehouse14.de` in prod, `http://localhost:3001`
  in dev): the single backend. The mobile app is a first-class client — same
  endpoints, same fiscal/TSE signing, same PIN auth as the desktop cashier.
- **packages/api-client**: the typed client SHARED by mobile + tauri-pos. One
  brain, two UIs.
- **packages/ui-kit**: the shared cashier tokens/components (mobile mirrors the
  exact tokens into `theme.ts` + `global.css` for NativeWind).

The mobile app does NOT pair with the desktop cashier over QR (that companion
system was removed — the native apps are now self-sufficient first-class
clients). Each platform authenticates independently via PIN and talks to the
server directly.
