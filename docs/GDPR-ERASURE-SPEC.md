# GDPR Art.17 Erasure — Definitive Build Spec (warehouse14)

## Model: ANONYMIZE-IN-PLACE, never row-delete

The entire system is built so the subject's identity rows (`customers`, `shoppers`) survive as anonymized shells; only PII *content* is scrubbed. Fiscal/GoBD/GwG rows are kept verbatim with embedded PII NULLed. Two cleanly-deletable surfaces exist (`shopper_sessions`). Everything else is scrub-in-place.

## Two entry keys
- **Dealer customer:** drive by `customers.id`.
- **Storefront shopper:** drive by `shoppers.id`, then ALSO run the full customer erasure on that shopper's `customers.customer_id` (1:1, `shoppers.customer_id` NOT NULL UNIQUE — `shoppers.ts:37-40`). Erasing a shopper without erasing its `customers` row leaves the KYC/PII master intact.
- **Phone-keyed sweeps (whatsapp / intake):** most of these rows have no customer FK. Resolve the subject's phone (decrypt `customers.phone_encrypted` or read `whatsapp_conversations.customer_phone_e164` where `customer_id` matches) and sweep by phone string. Intake has NO customer link at all — it must be driven by a separate session/phone retention sweep, not this transaction.

## Roles required
The app role (`warehouse14_app`) is deliberately barred from most of these writes. The erasure job MUST run as a **privileged owner role** (or `SECURITY DEFINER` functions) for: `customers` blind-index/encrypted cols where not granted, `kyc_documents` purge, `appointment_notifications.recipient`, all `whatsapp_*` raw-PII cols, `transactions.shipping_address_encrypted`, `transaction_items`, `products` intake-locked rows, `shoppers.email_encrypted`/`email_blind_index`, `mcp_tool_invocations`, `webhook_events`. Strictly-immutable tables (`ledger_events`, `tse_*`) reject ALL mutation even from owner via triggers — those are never touched.

## WHAT STAYS (and why it's lawful)
- **All fiscal/GoBD/GwG rows**: `transactions`, `transaction_items`, `transaction_payments`, `tse_*`, `ledger_events`, `daily_closings`, `dsfinvk_exports`, `vouchers`+`voucher_redemptions`, `kyc_documents` (as shells), `document_attachments` (fiscal categories, as shells). **One-line justification:** §147 AO / §257 HGB / GoBD impose a 10-year immutable retention that overrides Art.17 (Art.17(3)(b) GDPR — legal obligation); the records are kept, only PII *embedded* in them is redacted.
- **`customer_number`** and the anonymized `customers`/`shoppers`/`whatsapp_conversations` shell rows: kept for fiscal-join integrity and to make every FK-by-value reference still resolve (no cascade orphaning).

## KEY GOTCHA INDEX (the transaction must handle every one)
- NOT NULL → tombstone, never NULL: `customers.full_name_encrypted`, `shoppers.email_encrypted`/`email_blind_index`, `appointment_notifications.recipient`, `whatsapp_*.from_phone`/`to_phone`, `whatsapp_outbound.body`, `whatsapp_conversations.customer_phone_e164`, `document_attachments.r2_key`/`file_name`, `internal_tasks.title`.
- UNIQUE + NOT NULL → per-row token: `whatsapp_conversations.customer_phone_e164`, `intake_messages.whatsapp_message_id` (`'erased:'||id`).
- JSONB-object CHECK → set `'{}'`, never NULL: `whatsapp_inbound.raw_payload`, `payment_intents.outcome`, `webhook_events.payload`, `mcp_tool_invocations.arguments`/`result`(SUCCESS).
- Conditional-NOT-NULL CHECK → keep a placeholder, not NULL: `appraisals.rejection_reason` (REJECTED→'GELOESCHT'), `whatsapp_outbound.provider_error` (failed→stripped object), `internal_tasks.cancellation_reason` (CANCELLED→≥4 chars).
- Coupled all-or-nothing CHECK → set the whole group together: `kyc_documents` purge (5 cols + actor), `document_attachments` link CHECKs (keep `customer_id`), `internal_tasks` related-entity pair, `customers` trust↔note (reset `trust_level='NEW'`), `shoppers` has-credential (random hash + NULL google_sub).
- Ordering CHECKs → `soft_deleted_at` then `anonymized_at`, equal-or-after: `customers`, `shoppers`.
- Immutable-by-trigger → never in the write set: `tse_*`, `ledger_events`, `transaction_items`/`payments` (INSERT-only).
- Storno invariant → never NULL `transactions.customer_id`.
- Orphan-PII (no customer FK, missed by id-driven erasure) → separate session/phone sweep: `intake_sessions`, `intake_messages`, `intake_drafts`, `whatsapp_*` (phone-keyed), `webhook_events` (content-correlated).
