# Warehouse14 Desktop — Definitive Upgrade Plan

## State of the desktop (honest assessment)

The desktop is **not a prototype — it is a strong, mostly production-grade cashier with a thin governance shell bolted beside it**, and the distance to the goal is real but bounded. The `tauri-pos` money engine (bigint-cents, banker's rounding, per-line VAT, TSE sandwich on Verkauf, hardened checkout mutex) and the Rust command surface (keychain-held secrets, AES-256-GCM KYC vault, fidelity-first mocks, minisign updater) are genuinely mature. But three classes of problem stand between today and "complete cashier + ideal governance + working OTA":

1. **Live-till correctness/fiscal holes** — a silently-dropped mixed-tax-treatment guard (`cart-store.ts` `addLine` only returns `ALREADY_IN_CART`), Ankauf with no TSE signing, a TSE offline queue that is never drained and is wiped on sign-out (`AppShell.tsx` `PER_OPERATOR_STORAGE_KEYS` includes `warehouse14.tse-queue.v1`), `pos_intents` crash-recovery defined but written by **zero** call sites, and two fiscal-grade Rust bugs — `zvt_reverse_payment` (`commands/zvt.rs:211`) parses a Storno with a naive single `buf[6]/buf[7]` read instead of the authorize path's full conversation loop, and the `MAX_BCD_CENTS`/zero-amount guard exists only in the mock (`mock/zvt_mock.rs:34`), so the real authorize builds an over-large frame that is green in mock and malformed on the wire.
2. **Doctrine drift the mobile app already eliminated** — **89** raw `err.message`/`error.message`/`j.error.message` leaks across the two desktop apps (verified count; the audit's "74" undercounts), no shared German-text spine, raw enum/UUID tokens on the compliance panel, a wrong channel-flags model, and zero read-side offline resilience.
3. **An unfinished shipping story** — control-desktop has **no login front door at all** (`control-desktop/main.tsx` mounts `<App/>` session-less), the active release workflow (`.github/workflows/release.yml`) ships POS only with **zero** `control` references and no `VITE_API_BASE_URL`, macOS is ad-hoc-signed (`signingIdentity: "-"`), Windows is unsigned, versions diverge hard (tauri-pos `tauri.conf.json` **0.4.33** vs `package.json` **0.1.0** — stale by 40+ releases, and the updater compares the Tauri version), `plugin-updater` drifts (POS 2.10.1 vs control 2.5.1), and the env build trap is reproduced verbatim (`main.tsx:38` falls back to `localhost:3001`, `.env.local` present on the build host).

None of this blocks the till from selling today, but several items are compliance exposure on a live regulated shop.

### Top 5 risks

1. **Fiscal-record loss is live, not theoretical.** The TSE offline queue (`tse-service.ts`) is never drained and its localStorage key is nuked on sign-out; Ankauf is never TSE-signed at all; `pos_intents` crash recovery is unimplemented. A Fiskaly outage plus a sign-out permanently destroys KassenSichV evidence.
2. **The env build trap will ship a dead till.** `main.tsx:38` falls back to `localhost:3001`, `.env.local` exists on the documented build host, and the active workflow does not force the prod URL — a `tauri build` bakes a dead API into the `.dmg`/`.exe`, silently defeating honesty doctrine (b).
3. **The governance surface has no front door.** `control-desktop/main.tsx` mounts session-less, so every ADMIN read 401s on launch and there is no login screen — the most compliance-critical app is unusable cold.
4. **OTA is half-wired and unsigned.** The active workflow builds POS only, control's `latest-control.json` never exists, macOS is ad-hoc-signed and Windows unsigned — so "one big update" trips Gatekeeper/SmartScreen and control can never self-update.
5. **Doctrine (a) is broken on the most-watched screens.** Raw machine tokens reach humans on `KonformitaetPanel`, Aufgaben priority, the appraisal metal hint, and ~89 `err.message` sites — the exact class of leak mobile spent ~200 commits eliminating.

### How this revision differs from the draft

This version absorbs the skeptical review in full: it splits `audit-ui.ts` into a portable vocabulary layer plus a per-platform presentation layer (the draft's 0.1 would not have typechecked); pulls the CSP-`localhost` drop and control-desktop login **forward** and pushes the `plugin-updater` bump **back** to the OTA phase where it can actually be tested; clusters all **signed fiscal-record** work into Phase 1 and adds the two missing Rust fiscal ports (ZVT reversal loop, ZVT amount guard); makes the grep-gate count the **exact regex set the sweep fixes** rather than a hand-counted number; splits the three mega-steps (`1.3`, `1.4`, `5.7`); promotes four-eyes strong-auth to a named sub-step; adds an **erasure-completeness audit** covering the temp-PDF and new read-cache PII sinks; and hardens the OTA phase (canonical-version required check, prerelease audit of existing tags, hardened-runtime entitlements, shared-minisign-key verify).

## Phase 1 — Fiscal & money-path correctness, including the full signed record (P0)

**Goal:** close every fiscal correctness/loss hole so the till is safe to run through a Fiskaly outage, a sign-out, and a crash — **and** make the signed fiscal record complete (VAT breakdown + process-data + correct ZVT framing) in one adjacent cluster, because these change the **signed** bytes and must not ship half-done. Highest-severity phase; lands before any release.

**Exit criteria:** a mixed-treatment cart is rejected with a German toast; Ankauf produces a `tse_signatures` row; the TSE offline queue drains and survives sign-out; `pos_intents` is written before every fiscal network call and reconciled on startup; the signed TSE FINISH body carries the real per-VAT decomposition and process-data; ZVT reversal parses via the full loop and authorize rejects oversized amounts before framing.

- **1.1 — Restore the mixed-tax-treatment guard.** In `apps/tauri-pos/src/state/cart-store.ts` re-add the same-`taxTreatmentCode` check in `addLine` returning `MIXED_TAX_TREATMENT`; handle that tag in `Verkauf.tsx` with a clear German toast and release the just-made hold. **Doctrine:** correctness/fiscal. **Verify:** a `cart-store` test asserts a `§25a` + `STANDARD_19` add is rejected; live-check on the sim that the toast fires and the header treatment stays consistent.
- **1.2 — Bring Ankauf under the TSE sandwich.** Wrap `AnkaufBezahlenDialog.tsx` in the same `tse-service` INTENTION→FINISH sandwich as `BezahlenDialog.tsx`, then `recordTseSignature` against the returned `transactionId`, with the identical offline-queue fallback. **Doctrine:** governance / KassenSichV §146a. **Verify:** a fiscal integration test asserts an Ankauf finalize produces a `tse_signatures` row; live-check a buy-in on the sim shows a signed receipt.
- **1.3a — Migrate the TSE queue to the durable outbox.** Move `tse-service.ts` off `localStorage['warehouse14.tse-queue.v1']` into the Tauri-SQLite outbox (`outbox-store.ts`), and **remove that key from the sign-out purge in `AppShell.tsx` `PER_OPERATOR_STORAGE_KEYS`**. **Doctrine:** governance / GoBD §146, honesty. **Verify:** unit test that an enqueued obligation survives a sign-out (it is now in SQLite, not per-operator localStorage).
- **1.3b — Drain the TSE queue.** Add a periodic drain that replays each entry to the fiscal API + `recordTseSignature` and removes on success, with `in_flight` transition and an attempt cap. **Doctrine:** governance / GoBD §146. **Verify:** unit test the drain (enqueue → drain → row recorded → removed).
- **1.3c — Surface the queue + a spec'd overflow policy (product decision).** Surface a persistent Gerätemanager alert while the queue is non-empty. The overflow behavior is a **product decision, not a bare code change**: past a hard threshold, do **not** silently block sales (that turns a Fiskaly outage into a self-inflicted denial of service); instead require a step-up **"Manager-Freigabe: weiterverkaufen"** that is itself recorded in the audit ledger, and never roll off the cap. **Doctrine:** governance, honesty, availability. **Verify:** live-check the alert appears; live-check that past the threshold a manager step-up is required and logged, and the till still sells after approval.

- **1.4b — Startup reconcile routes into the outbox (names the dependency).** On startup, reconcile rows where `resolved_at IS NULL AND failed_at IS NULL` **into the durable outbox** (self-contained, Phase 1). Unreconcilable rows that need human judgement are flagged for the **Konfliktpostfach built in Phase 6.1** — until 6.1 lands they surface as a plain Gerätemanager count, so 1.4 fully lands in Phase 1 with no forward dependency for its verifiable core. **Doctrine:** correctness. **Verify:** integration test simulates a crash between confirm-tender and enqueue, restarts, and asserts the sale is recovered into the outbox.
- **1.5 — Populate the TSE VAT breakdown (signed body).** Extend `TseFinishParams` and `tse_finish_transaction` (`src-tauri/src/commands/tse.rs`) so `amounts_per_vat_id` carries the real per-VAT-rate cart totals instead of `[]`. **Doctrine:** governance / KassenSichV / DSFinV-K. **Verify:** a signed-body assertion test confirms the VAT decomposition is present for a mixed 19%/7%/§25a cart.
- **1.6 — Carry DSFinV-K process-data into TSE FINISH (signed body).** Carry the KassenSichV process-data into `tse-service.ts` FINISH and persist it on `tse_signatures` so `tse.csv` can emit the full signed payload. ## Checkpointed rollout strategy (never break the live till)

The live till (Intel iMac, `fiscal_active` per the go-live plan) must keep selling through every phase. Rules:

1. **Additive-first, behind the same green gate.** Every sub-step ends with `pnpm --filter @warehouse14/db build && pnpm -r typecheck` **plus** the Phase 0.3 desktop build. No sub-step merges without both green (doctrine c).
2. **Live-verify on the simulator, build only on command.** Iterate on the Mac-local sim per every sub-step's live-check; **do not build the `.ipa`/`.dmg`/`.exe` until Basel explicitly commands it** (build-on-command rule). Native-dep features (scale, keychain, Windows print, notarization) are the only build-phase exceptions, and those get a build only when the phase reaches its OTA rehearsal.
3. **Reversible ordering by risk.** Phase 0 changes nothing rendered (the `plugin-updater` behavior change was moved out to Phase 8). Phase 1 (fiscal) lands as isolated, individually-revertible commits with their own tests; the signed-body cluster (1.5–1.8) is co-located so the signed record never ships half-done. Parity (Phase 2) flips the grep-gate to failing **only after** the last leak is fixed, and each 2.1 file does body-replace-and-remove-`default:` in **one commit** with a per-commit typecheck so the exhaustive `Record` is always the only path — typecheck never goes red mid-sweep.
4. **P0 gaps are not schedule-buried.** Control-desktop login (5.1) is delivered **early, parallel to Phase 1**, or its broken window is documented. Four-eyes strong-auth (5.8) is a named sub-step, not a clause.
5. **The OTA cutover is the single riskiest moment — rehearse it on a throwaway tag first (8.9) before the real "big update" tag.** The port-prove-delete order (8.1) means the only working release path is never deleted before its replacement is proven. Keep `prerelease:false` discipline so `releases/latest` never resolves to an rc.
6. **No claim without evidence (doctrine d).** Each phase's exit criteria are the evidence contract: a passing test **and** a named live-check on the sim (or, for signing/OTA, an install on the real OS). External-blocked gates (Prüftool 6.6, device-revocation endpoint 5.7e) are tracked as blocked, never silently marked done.
